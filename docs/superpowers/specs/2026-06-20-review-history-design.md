# 복기 보관함 및 복기 이력 저장 설계

## 목표

사용자가 일반 복기 또는 심층 복기를 실행한 뒤, 이후에 당시 매매 차트와 AI 복기 내용을 함께 다시 볼 수 있게 한다. 기능 이름은 `복기 보관함`으로 분리하고, 매매복기 입력 화면과 과거 복기 열람 화면을 명확히 나눈다.

## 기본 방향

- `복기 보관함`을 별도 메뉴로 둔다.
- AI 복기 실행 결과는 로그인했고 `매매 이력 저장`을 켠 사용자에게만 자동 저장한다.
- 저장 기능이 꺼진 사용자는 현재처럼 1회성 결과만 보고, 저장하려면 저장 기능을 켜도록 안내한다.
- 저장된 복기 상세 화면은 당시 차트, 매수/매도 마크, AI 복기 내용, 사용된 복기 유형을 함께 보여준다.
- 복기 보관함 진입 시 전면 광고를 1회 노출한다. 보관함 안에서 여러 복기를 오가는 동안에는 추가 광고를 띄우지 않는다.

## 복기 유형별 저장 내용

### 일반 복기

- 단건 매매 중심으로 저장한다.
- 저장 항목:
  - 복기 유형: `basic`
  - 대상 매매 ID 또는 입력 당시 매매 스냅샷
  - 한 줄 총평
  - 잘한 점
  - 아쉬운 점
  - 다음 체크리스트 3개
  - 당시 차트 요약 데이터
  - 생성 시각, 사용 모델, 접근 차감 출처

### 심층 복기

- 이번 매매와 최근 5~10건의 매매 요약을 함께 저장한다.
- 심층 복기는 차트 기반 기술 분석을 포함한다.
- AI에는 차트 이미지를 매번 보내기보다, 백엔드가 계산한 기술적 요약을 전달한다.
- 저장 항목:
  - 복기 유형: `advanced`
  - 대상 매매 ID 또는 입력 당시 매매 스냅샷
  - 최근 매매 5~10건 요약
  - 반복 실수 분석
  - 손절 기준 분석
  - 진입 가설 분석
  - 대응 문제 분석
  - 매수 시점 기술 분석
  - 매도 시점 기술 분석
  - 다음 매매 규칙
  - 당시 차트 요약 데이터
  - 생성 시각, 사용 모델, 접근 차감 출처

## 차트 기반 기술 분석 방식

심층 복기에서 AI가 직접 차트 이미지를 보는 방식은 비용과 일관성 문제가 커질 수 있으므로 기본값으로 쓰지 않는다. 대신 서버가 차트 데이터를 먼저 분석하고, 그 요약을 AI에 전달한다.

서버가 계산할 항목:

- 매수/매도 시점의 종가 대비 체결가 위치
- 매수 전 5봉, 10봉, 20봉 수익률
- 매수 후 5봉, 20봉 흐름
- 거래량 20봉 평균 대비 체결일 거래량
- RSI 14
- 이동평균선 5, 20, 60, 120 위치
- 최근 고점/저점과 체결가의 거리
- 단타/스캘핑은 분봉, 스윙은 일봉 중심 차트 컨텍스트

AI에 전달할 내용은 원시 OHLCV 전체가 아니라 위 요약과 필요한 일부 캔들 구간으로 제한한다. 이렇게 하면 심층 복기 품질은 높이고 API 비용은 1회 호출 수준으로 유지할 수 있다.

## 복기 보관함 화면

목록 화면:

- 최근 복기 순으로 표시
- 종목명, 복기 유형, 매매 날짜, 손익, 생성 시각 표시
- 필터: 전체, 일반, 심층
- 검색: 종목명, 종목코드
- 삭제 버튼 제공

상세 화면:

- 상단: 종목명, 복기 유형, 매매 기간, 손익
- 중앙: 당시 차트와 매수/매도 마크
- 하단: AI 복기 내용
- 심층 복기인 경우 반복 실수, 손절 기준, 매수/매도 시점 기술 분석을 별도 섹션으로 표시

## 광고 정책

- 복기 보관함 메뉴에 처음 진입할 때 전면 광고를 1회 노출한다.
- 같은 앱 세션 안에서 복기 보관함 내부 이동에는 광고를 추가로 노출하지 않는다.
- 광고 로드 실패 또는 모바일 광고 SDK 미사용 환경에서는 보관함 진입을 막지 않는다.
- Pro 사용자 또는 광고 제거 상품 보유자는 복기 보관함 진입 광고를 보지 않는다.
- 이 광고는 AI 복기 실행을 위한 보상형 광고와 분리한다.
- 정책상 반복 광고와 예기치 않은 광고 노출을 피하기 위해, 화면 진입 버튼을 누른 직후의 자연스러운 전환 지점에서만 시도한다.

## 데이터 모델

새 SQLite 저장소 또는 기존 journal DB에 `review_history` 테이블을 추가한다.

필드 초안:

- `id`
- `user_id`
- `review_type`: `basic` 또는 `advanced`
- `ticker`
- `name`
- `trade_date`
- `target_trade_id`
- `trade_snapshot_json`
- `recent_trades_snapshot_json`
- `chart_snapshot_json`
- `ai_review_json`
- `access_snapshot_json`
- `model`
- `source`
- `created_at`
- `deleted_at`

초기 구현은 물리 삭제를 기본으로 한다. 이후 감사/환불/분쟁 대응이 필요하면 보존 정책을 별도로 설계한다.

## API

- `GET /api/journal/review-history`
  - 현재 로그인 사용자와 저장 기능이 켜진 사용자만 조회
  - 목록용 요약 반환
- `GET /api/journal/review-history/{id}`
  - 현재 로그인 사용자의 해당 복기 상세 반환
- `DELETE /api/journal/review-history/{id}`
  - 현재 로그인 사용자의 해당 복기 삭제
- `POST /api/journal/ai-review-once`
  - 기존 AI 복기 실행 후 조건이 맞으면 복기 이력을 저장하고 `review_history_id` 반환

## 개인정보와 내보내기/삭제

- 계정 데이터 삭제 시 복기 이력도 함께 삭제한다.
- 내 데이터 내보내기에는 저장된 복기 이력도 포함한다.
- AI 분석 원문은 저장 기능을 켠 사용자에게만 저장한다는 문구를 화면에 명확히 보여준다.

## 테스트 기준

- 저장 기능이 꺼진 사용자는 AI 복기를 실행해도 복기 이력이 저장되지 않는다.
- 저장 기능이 켜진 로그인 사용자는 AI 복기 후 복기 이력이 저장된다.
- A 사용자 이력은 B 사용자에게 보이지 않는다.
- 복기 이력 상세에는 차트 스냅샷과 AI 복기 내용이 함께 반환된다.
- 계정 데이터 삭제 시 복기 이력도 삭제된다.
- 내 데이터 내보내기에 복기 이력이 포함되고 세션 토큰은 포함되지 않는다.
- 복기 보관함 진입 광고는 세션당 1회만 시도된다.

## 구현 순서

1. 백엔드 복기 이력 저장소와 테스트 추가
2. AI 복기 실행 결과 저장 연결
3. 계정 데이터 삭제와 내보내기에 복기 이력 포함
4. 프론트 복기 보관함 목록/상세 화면 추가
5. 복기 보관함 진입 광고 정책 추가
6. 수동 테스트 가이드와 개발 이력 업데이트
