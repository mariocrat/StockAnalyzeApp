# AI 복기 동의 이력 저장 설계

## 목적

AI 복기 실행 전에 사용자가 개인정보 및 매매 기록 전송에 동의했다는 사실을 계정에 기록한다. 이미 계정 테이블에는 `privacy_consent_version`, `privacy_consented_at` 컬럼이 있으므로, 새 저장소를 만들지 않고 기존 계정 저장소를 사용한다.

이 기능은 사용자가 나중에 내 데이터 내보내기를 했을 때 동의 이력을 확인할 수 있게 하고, 운영 배포 전 개인정보처리방침/Google Play 심사 대응에 필요한 최소 기록을 남기는 것이 목표다.

## 범위

- `POST /api/journal/ai-review-once`에서 `privacy_consent: true`가 확인되면 현재 로그인 계정의 동의 버전과 동의 시각을 저장한다.
- 동의 버전은 서버 상수 또는 환경변수로 관리한다. 기본값은 `ai-review-privacy-v1`로 시작한다.
- 로그인하지 않은 개발용/일회성 호출은 AI 접근 검증은 기존처럼 동작하되, 계정 동의 이력은 저장하지 않는다.
- 계정 데이터 내보내기에는 이미 `user` 객체가 포함되므로, 동의 버전과 시각이 자연스럽게 포함된다.
- 매매복기 화면의 계정 관리 영역에 동의 상태를 표시한다.

## 비범위

- 개인정보처리방침 전문 작성이나 법률 검토는 이번 작업에 포함하지 않는다.
- 동의 철회 UI는 이번 작업에 포함하지 않는다. 계정 데이터 삭제와 매매 이력 저장 OFF가 현재 사용자가 선택할 수 있는 삭제/비저장 수단이다.
- 과거 동의 버전 변경 이력을 여러 건 쌓는 감사 로그 테이블은 만들지 않는다. 초기 배포 전 단계에서는 계정의 최신 동의 상태만 저장한다.

## 데이터 흐름

1. 사용자가 매매복기 화면에서 AI 분석 동의 체크박스를 켠다.
2. 프론트엔드는 기존처럼 `privacy_consent: true`를 `POST /api/journal/ai-review-once`에 보낸다.
3. 서버는 `verify_ai_review_access()`로 동의 여부와 이용권/광고/구독 접근 권한을 검증한다.
4. 검증이 통과하고 로그인 세션이 있으면 `users.privacy_consent_version`, `users.privacy_consented_at`을 갱신한다.
5. 이후 `GET /api/me/data-summary`, `GET /api/me/export-data`, 로그인 응답의 `user` 객체에서 최신 동의 상태를 확인할 수 있다.

## 화면 표시

계정 관리 영역의 데이터 요약에 `AI 동의 기록` 항목을 추가한다.

- 동의 기록이 있으면 `동의 완료`와 날짜를 표시한다.
- 동의 기록이 없으면 `미기록`으로 표시한다.
- 버튼이나 긴 설명은 추가하지 않는다. AI 복기 실행 직전 동의 체크박스가 실제 동의 액션이다.

## 오류 처리

- `privacy_consent`가 false이면 기존처럼 `400 Privacy consent is required before AI analysis.`를 반환한다.
- 동의 이력 저장 실패가 생기면 AI 복기 실행을 실패시키는 편이 맞다. 동의 기록 없이 AI 분석이 진행되는 상태가 더 위험하기 때문이다.
- 로그인하지 않은 일회성 분석에서는 저장할 계정이 없으므로 동의 이력 저장을 건너뛴다.

## 테스트

- 계정 저장소 단위 테스트: 동의 버전/시각을 저장하고 현재 세션 사용자에게 반영되는지 확인한다.
- AI 복기 라우트 테스트: 로그인 사용자가 동의 후 AI 복기를 실행하면 계정 동의 이력이 갱신되는지 확인한다.
- 동의 누락 테스트: `privacy_consent: false`이면 기존 접근 검증이 실패하는지 유지한다.
- 프론트 검증: ESLint와 Vite 빌드가 통과해야 한다.
